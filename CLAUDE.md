# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repo layout

This is a `uv` workspace with two Python packages plus a Docusaurus site:

- `src/bank2ai/` — the published `bank2ai` library (PyPI). Holds the Pydantic models (`models.py`) and the FastMCP tool surface (`tools.py`, exposes `register_tools`).
- `examples/demo/` (`bank2ai-demo`) — reference MCP server backed by hardcoded data in `data.py`. Used as the source-of-truth implementation when generating the language-neutral spec.
- `specs/` — language-neutral spec: `bank2ai.spec.md` (narrative) and `bank2ai.json` (JSON Schema, autogenerated).
- `scripts/generate_spec.py` — regenerates `specs/bank2ai.json` from the demo server's registered tools and shared Pydantic models.
- `docs/` — Docusaurus site for [bank2ai.com](https://bank2ai.com) (separate `package.json`, uses `yarn`).


## Commands

All Python commands run via `uv` from the repo root. Package selection uses `--package <name>`.

```bash
# Install/sync the workspace
uv sync

# Run the demo MCP server (stdio)
uv run --package bank2ai-demo bank2ai-demo

# Run the demo's interactive client against the demo server
uv run --package bank2ai-demo python -m bank2ai_demo.client

# Demo tests (includes the spec drift test)
uv run --package bank2ai-demo pytest
uv run --package bank2ai-demo pytest tests/test_schema_sync.py::test_committed_spec_matches_generator

# Regenerate specs/bank2ai.json after touching tools.py or models.py
uv run python scripts/generate_spec.py

# Docs site (separate toolchain)
cd docs && yarn && yarn start
```

## Architecture

**The library exposes a tool *spec*, not a server.** `bank2ai.register_tools(app, ...)` decorates a `FastMCP` app with the canonical bank2ai tools. Each tool is keyed by an optional async handler kwarg (`get_accounts=`, `get_transactions=`, …); tools whose handler is omitted are simply not registered, so a server can expose any subset of the surface. Handlers receive snake_case kwargs matching the tool's input schema and return Pydantic models (or dicts shaped like them); FastMCP infers the output schema from the response-model annotation on the wrapper.

**The Python package is currently the source of truth for the spec.** `scripts/generate_spec.py` imports the demo server's `app`, calls `app.list_tools()`, strips FastMCP vendor extensions (`x-fastmcp-*`), serializes the four documented Pydantic models (`Account`, `Transaction`, `Category`, `Recipient`), and writes `specs/bank2ai.json`. The `test_committed_spec_matches_generator` test in `examples/demo/tests/test_schema_sync.py` runs `build_spec()` in memory and asserts equality with the committed file — so any change to `tools.py` or the four documented models requires re-running the generator. The drift test will fail CI otherwise.

**Spec versioning is independent of the package.** `SPEC_VERSION` lives in `scripts/generate_spec.py` and follows SemVer per the rules in `specs/README.md` (major = breaking tool/field changes, minor = additive, patch = description-only). The `bank2ai` package version in `pyproject.toml` is separate.

**Tool naming.** Wire-level tool names are kebab-case (`get-accounts`, `prepare-transfer`, `get-recipients`); handler kwargs are snake_case, and Pydantic model field names on the wire are camelCase. The `register_tools` adapter bridges kwargs to the wire.

## Conventions

- `bank2ai` is always lowercase in prose; `Bank2ai` only at sentence start. Never `Bank2AI`.
- After editing tools or the four documented models, regenerate the spec — don't hand-edit `specs/bank2ai.json`.
